# 2단계 계획서 — 필터 기반 캠핑기어 파인더

> 작성: 2026-06-09 · 전제: 1단계 완료(verified 2,324종·별점·LIMITS, `STAGE1-REPORT.md`)
> 한 줄 목적: **사용자가 다양한 필터를 걸어, 자신에게 필요한 제품을 빠르게 찾는다.**

---

## 1. 목적과 성공 기준

비교표가 아니라 **파인더(finder)**다. 핵심 가치는 "내 조건에 맞는 후보를 몇 초 만에 좁히기".

**성공 기준**
- 사용자가 카테고리 진입 후 **3~5번의 필터 조작**으로 후보를 한 화면에 좁힌다.
- 필터는 **즉시 반영**(클릭과 동시에 결과·개수 갱신, 서버 왕복 0).
- 결과의 모든 별점·값에 **신뢰 배지**가 붙어, "이 추천이 왜 믿을 만한지"가 보인다.
- 데이터 한계(미공개 지표)는 숨기지 않고 **필터로 다룰 수 있다**(예: "용량 데이터 있는 코펠만").

---

## 2. 대표 사용자 시나리오 (필터 동선)

1. **백패킹 입문자** — `백패킹텐트` → 인원 `2인` → 무게 `★4+` → 가격 `~30만원` → 결과 8종 → 무게순 정렬
2. **차박러** — `오토캠핑텐트` → 인원 `4인` → 바닥면적 `≥9㎡` → 내수압 `≥2000mm` → 브랜드 `코베아·스노우피크`
3. **겨울 캠퍼** — `침낭` → 내한온도 `≤ -10℃` → 충전량 `데이터 있는 것만` → 무게순
4. **무게 민감 사용자** — `의자` → 무게 `★4.5+` → 내하중 `≥100kg` → 가격순
5. **가성비 탐색** — 카테고리 무관 → `가격 ≤ X` + `종합 별점 높은 순`

→ 공통 패턴: **카테고리 선택 → 인원/스펙/가격/별점/브랜드 필터 → 정렬 → 좁혀진 결과**.

---

## 3. 정보 구조 (IA)

```
허브(index)            카테고리 16개 + 등급배지 + "무엇을 찾으세요?" 진입
   └ 파인더(category)   ← 2단계의 핵심 화면
        ├ 좌: 필터 패널(스펙 슬라이더·브랜드·인원·가격·별점·품질)
        ├ 우: 결과 리스트(정렬 가능, 별점+배지, 활성필터 칩)
        └ (선택) 제품 상세/비교 담기
```

- 모바일: 필터는 상단 "필터" 버튼 → 바텀시트.
- 카테고리마다 지표가 달라, 필터 패널은 **해당 카테고리 metrics에서 동적으로 생성**.

---

## 4. 필터 설계 (이 단계의 핵심)

필터 종류는 1단계 데이터 구조에서 자동 도출된다.

| 필터 | 소스 | UI | 비고 |
|---|---|---|---|
| **카테고리** | categories | 허브 카드 / 셀렉트 | 진입점 |
| **인원** | products.capacity | 칩(1·2·3·4인…) | 텐트류만 |
| **가격대** | canonical min/max(국내우선) | 듀얼 슬라이더 | 천원 단위 |
| **스펙 범위** | ★지표 value_normalized | 지표별 듀얼 슬라이더 | 예: 무게 ≤ X g, 내수압 ≥ X mm |
| **별점 임계** | ratings.stars | "★4+ / ★4.5+" 토글 | 지표별 |
| **브랜드** | brands | 멀티 체크(검색) | 인기순/가나다 |
| **데이터 품질** | source_id·star_eligible·valid | 토글 | "확정값만" / "데이터부족 제외" / "외형기준 제외" |
| **텍스트 검색** | brand+model | 검색창 | 즉시 필터 |

**방향 인지.** weight·comfort_temp는 *낮을수록 좋음* → 슬라이더 라벨을 "최대 무게"로. water_head·floor·capacity 등은 *높을수록 좋음* → "최소".

**Facet 카운트.** 각 필터 옵션 옆에 해당 결과 수 표시(예: `코베아 (23)`)로 "빈 결과" 막다른 길 방지.

**정직성을 필터로.** 한계 지표(R값·화력 등)는 기본적으로 "데이터부족 포함"이되, *그 지표로 정렬/필터하면* 자동으로 "값 있는 것만"으로 좁혀지고 표본 수를 경고한다.

---

## 5. 정렬 & 결과 표시

- **정렬**: 가격 / 각 지표 값 / 각 지표 별점 / (선택)종합 — 오름·내림 토글. 기본 = 그 카테고리 대표 지표 별점 내림차순.
- **결과 행**: 브랜드·모델(변형 통합 +N색)·인원·가격범위·지표별(값+★+배지).
- **활성 필터 칩**: 상단에 현재 조건 칩으로 표시, 개별 ✕로 해제, "전체 초기화".
- **결과 수 / 빈 결과**: "N개 모델" 상시 표시, 0이면 "필터를 완화하세요" + 가장 강한 제약 제안.
- **비교 담기(선택)**: 체크한 2~4개를 나란히 비교(후속).

---

## 6. 기술 설계 (정적 유지)

- **여전히 서버리스.** 데이터(2,165모델)는 작아서 **전부 클라이언트에서 필터링** → 즉시성 확보, 배포 단순.
- 카테고리 JSON 1개를 받아 메모리에서 필터/정렬(현 `app.js` 구조 확장).
- 프레임워크 없이 바닐라 JS 유지(번들·빌드 불필요). 상태는 **URL 쿼리스트링에 직렬화**(`?cat=tent&cap=2&w_max=2000&star_weight=4`) → 공유·뒤로가기 가능.
- 배포: **GitHub Pages**(정적 그대로). `site/`를 publish.

**`export_site.py` 보강(2단계용 데이터 추가)**
- 카테고리별 각 지표의 **min/max/분포**(슬라이더 경계용)
- **브랜드 목록 + 카운트**(facet)
- 인원 옵션 목록
- 가격 min/max
- (이미 있음) 모델별 값·별점·배지

## 8. 구현 로드맵 (마일스톤)

- **M0 (완료)** — MVP: 허브 + 카테고리 비교표 + 별점/배지 + 정렬/검색 (`site/`)
- **M1 필터 엔진** — `export_site.py`에 facet/min-max 추가 → 필터 패널(인원·가격·스펙 슬라이더·별점) 클라이언트 필터링 + facet 카운트
- **M2 상태·공유** — 필터 상태 URL 직렬화, 활성필터 칩, 빈결과 처리, 초기화
- **M3 품질 필터 + 모바일** — 확정값만/데이터부족 제외/외형기준 제외 토글, 한계 경고 연동, 반응형 바텀시트
- **M4 비교 담기 + 상세** — 2~4개 나란히 비교, 모델 상세(전 지표·가격이력·확정근거)
- **M5 다듬기·배포** — 디자인 폴리시, 접근성, GitHub Pages 배포

---

## 9. 비목표(2단계에서 안 함)

- **수익화(쿠팡 파트너스 링크)** — 3단계. 1·2단계 완벽 후. (제품행에 링크 슬롯만 미리 비워둠)
- 사용자 계정·리뷰·찜 — 정적 사이트 범위 밖
- 실시간 가격 크롤링 — 데이터는 배치 갱신(파이프라인 재실행)

---

## 10. 3단계 연결 고리(미리 메모)

- 결과 행/상세에 **제휴 링크 슬롯**을 데이터 모델에 예약(빈 필드)해 두면, 3단계에서 채우기만 하면 됨.
- 정직성(배지·한계)이 곧 신뢰 → 전환의 토대. 수익화는 신뢰를 훼손하지 않는 선에서.

---

**요약:** 2단계는 1단계의 신뢰 데이터를 **"빠르게 좁히는 필터 경험"**으로 바꾸는 작업이다. 서버 없이, 즉시 반응하는 파인더. MVP(M0)는 이미 섰고, 다음은 **M1 필터 엔진**부터다.
