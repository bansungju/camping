> 성격: **공통(cross-cutting) 인증 정책 + 프론트 토대**. 코드 미수정, 기획안.

## 1. 한 줄 요약
개인화/로그인 필요 기능은 **실행 직전에 단 하나의 공용 노드 `requireLogin()`** 를 통과해야 한다. 비로그인이면 **현재 동선·의도를 저장한 뒤 로그인 페이지로 보내고, 로그인 후 원래 자리/액션으로 복귀**시킨다. 페이지 어디서나 로그인 상태를 알 수 있게 **전역 auth 토대**를 깐다.

> **정책 결정(2026-06-13): "완전 로그인 강제".** 찜·세트도 로컬 우선을 중단하고 로그인 게이트 적용.
> **전제(2026-06-13):** 아직 **상용 배포 전 · 전원 테스트 사용자**라 기존 익명 localStorage 데이터는 **날려도 됨** → 게스트 데이터 마이그레이션/병합 불필요(설계에서 제외).

## 2. 배경 — 현황 진단 (코드 근거)

### 2-1. 지금은 기능이 두 부류로 갈려 있다
| 부류 | 기능 | 로그인 체크 | 비로그인 동작 | 근거 |
|------|------|------------|---------------|------|
| **로컬 우선(게이트 없음)** | 찜, 내 세트 | ❌ | localStorage 저장 후 로그인 시 동기화 | `toggleWish`([app.js:309](site/app.js)), `addToSet`/`newSet`([app.js:354](site/app.js)) |
| **로그인 게이트(있으나 제각각)** | 후기 | ✅ UI+제출 2중 | 폼 숨김+힌트 | `wireReviews`([app.js:2119](site/app.js)) |
| | 커뮤니티 글/댓글/좋아요 | ✅ | 리다이렉트/폼숨김/alert | `canParticipate`([community.html:147](site/community.html)) |
| | 닉네임/회원탈퇴 | ✅ | 로그인 후에만 노출 | `account.html` |

### 2-2. 진짜 문제 2가지 (사용자 지적의 핵심)
- **G1. 공용 게이트 노드 부재.** `requireLogin()`/`ensureAuth()` 같은 단일 진입점이 없어 기능마다 **토스트·alert·폼숨김·리다이렉트**가 제각각 → 경험 불일치, 누락 위험.
- **G2. 전역 로그인 상태 부재.** `window._accUser`가 **`account.html`에서만** 세팅됨([account.html:543](site/account.html)). 카테고리·상세 등 다른 페이지에선 로그인 여부를 신뢰성 있게 알 수 없어, **일관된 게이트를 만들 토대 자체가 없음.**

> 결론: 막연히 "전부 막자"가 아니라 **토대(G2) + 노드(G1)** 를 먼저 세우고, 그 위에 정책(완전 강제)을 적용한다.

## 3. 핵심 설계

### 3-1. 전역 auth 토대 (G2 해소)
- auth 초기화를 `account.html` 전용에서 **모든 페이지 공통 부트스트랩**으로 이동(`app.js`에서 `initAuth` 1회).
- 노출 API:
  - `authReady(): Promise<void>` — `getSession()` 완료 보장(게이트가 await).
  - `isLoggedIn(): boolean` — 준비 후 동기 판별.
  - `currentUser()` — 사용자 객체(또는 null). 전역 단일 진실(`window._accUser` 대체/승격).
  - `onAuthChange(cb)` — 로그인/로그아웃 구독(헤더·버튼 상태 갱신).

### 3-2. 공용 게이트 노드 `requireLogin()` (G1 해소) — 단일 진입점
```
async function requireLogin({ action, returnTo } = {}) {
  await authReady();
  if (isLoggedIn()) return true;
  savePendingIntent({ action, returnTo: returnTo ?? location.href }); // sessionStorage
  goToLogin();                 // 로그인 페이지로 이동(또는 인터셉트 모달→로그인)
  return false;
}
```
- **모든 로그인 필요 액션은 이 노드를 먼저 호출**하고, `false`면 즉시 중단(이미 리다이렉트됨).
- 후기·커뮤니티·계정의 기존 개별 체크도 **이 노드로 통일**(중복 제거).

### 3-3. 동선·의도 보존 (로그인 후 복귀/재개)
- 게이트 진입 시 **의도(intent)** 저장: `{ action, params, returnTo }` → `sessionStorage`.
- 로그인 콜백(`auth-callback.html`) 처리 후:
  1. `returnTo`로 복귀(원래 페이지).
  2. 저장된 `action`이 있으면 **그 액션을 재개**.
- **자동 재개 범위(결정 2026-06-13):**
  - **찜 토글 · 세트 담기 → 자동 재개.** 돌아오자마자 시스템이 실행(찜은 토글, 세트는 담기 모달 재오픈+담기). 가장 빈번·쉬운 둘만 매끄럽게.
  - **후기 → 자동 재개 없음.** 대신 **폼을 여는 진입점에서 미리 게이트**(아래 원칙). 로그인 후 폼만 열림, 작성 내용 보존 불필요.
- **원칙 — 게이트는 가능한 한 일찍(진입점에서).** 폼/입력을 시작한 뒤 중간에 인터셉트해 내용을 날리지 말고, **동작을 시작하기 전**에 로그인을 요구한다(후기 폼 오픈 전, 글쓰기 진입 전 등).

### 3-4. 일관 UX (인터셉트 표준화) — 1-스텝 모달 확정
> 결정(2026-06-13): **1-스텝 인터셉트 모달**(즉시 리다이렉트 대신). 맥락 없이 갑자기 튕기는 것 방지.
- 비로그인 클릭 시 **표준 모달** 1개: "이 기능은 로그인이 필요해요 · 로그인하면 기기 간 동기화돼요 [로그인하기]". 한 번 탭 → 로그인 페이지.
- 흩어진 토스트/alert/폼숨김 문구를 **이 표준 하나로 대체**. 카피·디자인 1벌로 관리.

### 3-5. 게스트 데이터 — 마이그레이션 불필요 (전제상 제외)
> **상용 배포 전 · 전원 테스트 사용자** 전제(§1) → 기존 익명 localStorage 데이터는 **버려도 됨**. 별도 병합/유실 방지 설계 불필요.
- 비로그인 상태에서 쌓인 로컬 찜·세트는 **무시 또는 비로그인 진입 시 정리**. 로그인 후 localStorage는 **로그인 사용자 캐시**로만 사용.
- (상용 출시 직전 사용자 데이터가 실제로 생기는 시점엔 병합 설계를 다시 검토 — 그때 별도 항목으로.)

| 기능 | 게이트 지점(엔트리) | 현재 | 변경 |

## 5. 표준 플로우

```
[개인화 액션 클릭]
      │
      ▼
 requireLogin() ── authReady() 대기
      │
      ├─ 로그인됨(Y) ──────────────▶ 액션 실행
      │
      └─ 비로그인(N)
             │ intent 저장(action+returnTo)
             ▼
        [인터셉트: "로그인이 필요해요"] ──▶ 로그인 페이지
             │
             ▼  (OAuth 완료, auth-callback)
        게스트 데이터 병합 업로드
             │
             ▼
        returnTo 복귀 + 저장된 action 재개
```

## 6. 엣지 케이스
- **로그인 도중 이탈/취소:** intent는 sessionStorage라 탭 종료 시 정리. 원래 페이지 그대로.
- **auth 준비 전 빠른 클릭:** `authReady()` await로 레이스 방지(로그인됐는데 게이트가 잘못 튕기는 일 차단).
- **로그아웃 상태에서 다른 탭 로그인:** `onAuthChange`로 버튼/헤더 상태 갱신.
- **재개 불가 액션:** 최소 복귀만 보장(액션은 1회 재시도).

## 7. 단계적 적용
| 단계 | 범위 |
|------|------|
| **A. 토대** | 전역 auth 부트스트랩 + `authReady`/`isLoggedIn`/`currentUser` + `requireLogin` 노드 + 표준 인터셉트 |
| **B. 일원화** | 후기·커뮤니티·계정의 기존 개별 체크를 공용 노드로 교체(동작 동일, 코드 정리) |
| **C. 강제 확대** | 찜·세트에 게이트 적용(테스트 데이터라 마이그레이션 없음) |
| **D. 동선 재개** | returnTo 복귀 + **찜·세트 담기 자동 재개**(후기는 진입점 게이트로 해결) |

---

## 8. 수용 기준 (AC)
1. 모든 페이지에서 로그인 상태를 신뢰성 있게 판별한다(`isLoggedIn` after `authReady`).
2. 로그인 필요한 모든 액션이 **동일한 인터셉트**를 거친다(문구·동선 일관).
3. 비로그인 클릭 시 의도가 저장되고, 로그인 후 **원래 위치로 복귀**한다(가능한 액션은 재개).
4. 읽기 동선(열람·검색)은 로그인 없이 그대로 동작한다.
