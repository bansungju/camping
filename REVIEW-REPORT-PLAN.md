# 후기 신고 버튼 — 상품 후기(reviews) 신고 동선 기획서

> 작성: 2026-06-13 / 작성자: 기획
> 대상: 상품 상세 모달의 유저 후기(reviews)
> 성격: **프론트 동선 추가**. 백엔드(`reports` 테이블·`reportContent`)는 이미 존재 → 신고 버튼 UI/이벤트만 추가.
> 관련: `app.js`(후기 모듈), `supabaseClient.js`(`reportContent`), `community.html`(기존 신고 패턴 참조)

---

## 1. 한 줄 요약
현재 사용자에게 **활성 노출**되는 유일한 UGC인 상품 후기에 **신고 수단이 없다.** 후기 상세 라이트박스에 `🚩 신고` 버튼을 추가해, 기존 `reportContent({ target_type: 'review', ... })` 백엔드로 연결한다.

> **레이어 구분(중요):** "후기는 비로그인 포함 누구나 **본다**"는 *왜 신고가 필요한가*(악성 후기 도달 범위가 넓음)의 근거일 뿐, *누가 신고하나*와는 별개다. **신고 행위 자체는 로그인 필수**다 — 익명 무한 신고를 막고 책임을 부여하기 위함이며, 커뮤니티 신고(`로그인 후 신고할 수 있어요`)와 동일한 정책이다.

---

## 2. 배경 — 현황 진단 (코드 근거)

### 2-1. 신고 기능은 영역마다 갈려 있다
| 영역 | 신고 버튼 | 백엔드 | 상태 |
|------|----------|--------|------|
| 커뮤니티 글 | ✅ `🚩 신고`([community.html](site/community.html)) | `reportContent({target_type:'post'})` | 커뮤니티 **비활성** → 도달 불가 |
| 커뮤니티 댓글 | ❌ 없음 | (범용 지원) | — |
| **상품 후기** | ❌ **없음** | (범용 지원) | **기능 활성** · 신고만 누락 |

### 2-2. 핵심 문제
- **후기는 비로그인 포함 누구나 읽는 활성 기능**인데(상품 상세 모달), 욕설·허위/광고성 후기·타인 비방을 사용자가 신고할 수단이 전혀 없다.
- 백엔드는 이미 준비됨: `reportContent({ target_type, target_id, reason })`([supabaseClient.js:207](site/supabaseClient.js))는 `target_type`을 범용으로 받아 `reports` 테이블에 적재. 커뮤니티 글이 `'post'`를 넘기듯, 후기는 **`'review'`만 넘기면 됨.**
- 즉 **신규 마이그레이션·백엔드 작업 불필요.** 프론트 버튼 + 이벤트만 추가하면 끝.

---

## 3. 핵심 설계

### 3-1. 배치 위치
**후기 상세 라이트박스(`openReviewDetail`, [app.js:2179](site/app.js))** 하단.
- 후기 카드(`_reviewCard`)에는 넣지 않음 — 그리드가 조밀해 오작동·시각 노이즈 우려. **상세를 연 뒤** 신고하는 동선이 자연스럽고 오신고도 줄인다.
- 위치: 닫기(✕) 외에, 본문(`.pmrvd-body`) 끝에 작은 텍스트 버튼. 커뮤니티 `.cm-report`와 동일 톤(`color:var(--faint)`, 작은 글씨, 우측 정렬).

### 3-2. 동작 플로우 (커뮤니티 신고와 일관)
1. `🚩 신고` 클릭
2. **로그인 확인** — 비로그인이면 `로그인 후 신고할 수 있어요.` 안내(토스트/alert). (로그인 게이트 기획 적용 시 `requireLogin()`으로 일원화 → [AUTH-GATING-PLAN.md](AUTH-GATING-PLAN.md))
3. **본인 후기 예외** — 자기 후기엔 신고 버튼 미노출(작성자 `user_id === currentUser().id` 비교). 커뮤니티가 본인 글에 삭제만 노출하는 것과 동일.
4. **사유 입력** — `신고 사유를 적어주세요 (5자 이상)`. 5자 미만/취소 시 중단.
5. `reportContent({ target_type: 'review', target_id: r.id, reason })` 호출.
6. 결과 안내 — 성공 `신고가 접수되었어요. 검토 후 조치합니다.` / 실패 `getErrorMessage(error)`.

### 3-3. 중복·악용 방지
- `reports` 테이블에 동일 `(reporter_id, target_type, target_id)` **유니크 제약이 있는지 확인** → 있으면 중복 신고 시 친절 메시지(`이미 신고한 후기예요.`). 없으면 후속 백엔드 과제로 분리(이번 범위 밖, 아래 §5).
- 신고 후 버튼을 `신고됨`으로 비활성화(같은 세션 재클릭 방지, UX 보조).

## 4. 수정 범위

| 파일 | 변경 |
|------|------|
| `app.js` | `openReviewDetail()` 렌더 HTML에 신고 버튼 추가 + 클릭 핸들러(로그인·본인·사유·`reportContent`). 후기 객체에 `user_id` 포함되는지 select 컬럼 확인([app.js:2155](site/app.js) 부근). |
| `style.css` | 신고 버튼 톤 클래스(커뮤니티 `.cm-report` 재사용 또는 동등 스타일) |
| (백엔드) | **변경 없음** — `reportContent`·`reports` 테이블 그대로 사용 |

---

## 5. 제외 범위 (후속 분리)
- 커뮤니티 댓글 신고(`target_type:'comment'`) — 커뮤니티 재활성화 시 함께.
- `reports` 중복 신고 유니크 제약·신고 누적 자동 숨김(임계치) 등 **운영/모더레이션 백엔드** — 별도 마이그레이션 필요.
- 신고 처리 관리자 화면 — 본 기획 범위 밖.

## 6. 검증 (배포 후, 라이브 gear-forest.com 기준)
- 타 사용자 후기 상세 → `🚩 신고` 노출 / 본인 후기엔 미노출
- 비로그인 클릭 시 로그인 안내
- 5자 이상 사유 제출 → Supabase `reports`에 `target_type='review'` 행 적재 확인
- 수정 후 `stamp_version.py` 실행(캐시버스팅) 필수
