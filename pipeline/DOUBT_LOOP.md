# 적대적 의심-보완 루프 (15분, 번갈아)

목적: **서브에이전트가 데이터를 의심(공격) → 메인이 검증·보완(방어)**. 자기검증 사각지대 제거.
작업폴더 ~/idea/camping-gear-app, DB=camping_tents500.db. 핸드오프 파일: `doubt_queue.md`.

## 사이클 판정 (큐 상태로 자동 번갈아)
매 회차 `doubt_queue.md`를 본다:
- **OPEN 의심이 있으면 → [보완 사이클] (메인)**
- **OPEN이 없으면 → [의심 사이클] (서브에이전트 호출)**

→ 의심이 큐 채움 → 다음 회차가 비움(보완) → 다음 회차가 다시 채움 → 자연 교대.

## [의심 사이클] — 새 서브에이전트 호출
`Task(subagent_type='general-purpose')` 로 **새 서브에이전트**를 띄워 지시:
> "~/idea/camping-gear-app 의 camping_tents500.db 를 적대적으로 감사하라. 지금까지 값에서 의심스러운 것을 찾아라:
> ① 전 지표 극단값 스캔(min/max)에서 물리적으로 이상한 값 ② 단위 혼입(ml/L, kcal/W, g/kg) ③ 기준 불명/혼입(무게 min·packed, 내한온도 comfort·limit) ④ 오분류(카테고리 안 맞는 제품) ⑤ 별점 역전/이상 ⑥ 크로스소스 pcode 오매핑.
> 발견 의심 5~10개를 근거(쿼리결과)와 함께 반환하라. 이미 알려진 것(capacity_l 단위=수정완료)은 제외. 측정값 기준, 추측 금지."
- 서브 반환을 `doubt_queue.md`에 `## OPEN [날짜] <한줄요약>` + 근거로 추가.

## [보완 사이클] — 메인이 처리
OPEN 의심 각각을:
1. **검증**: 직접 쿼리로 사실 확인.
2. **판정**: 진짜 문제면 → 수정(파서/타당범위/크로스소스/분류). 아니면 → 근거와 함께 **기각**.
3. **표시**: 해당 항목을 `RESOLVED(수정)` 또는 `REJECTED(근거)` 로 변경.
4. 수정했으면 `python3 pipeline/validate_ranges.py` + `run_all.py` 재집계.
5. `doubt_log.md`에 처리 내역 기록(맨 위 한줄요약 갱신).

## 원칙
- 측정값만, 추측 금지, 멱등.
- 서브에이전트는 **공격만**(의심 제기), 메인은 **방어**(검증·수정). 역할 분리가 핵심.
- 진짜 문제는 고치고, 아닌 건 명확히 기각해 큐를 비운다.
